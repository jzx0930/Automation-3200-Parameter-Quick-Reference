PSO Specifications

Hardware: PSO does not apply when used with the Nmark SSaM or the Nstep.

The hardware that you have in your system has an effect on the PSO operation. The specifications that follow tell you how the PSO operation is limited by different drive types and feedback options.

Maximum PSO Encoder Input Data Rates

The table that follows shows the maximum permitted PSO input encoder data rates for correct operation. If the input to the PSO exceeds this rate, incorrect tracking and pulse generation will occur. The PSO hardware contains scaling functions (refer to PSOTRACK SCALE command) which let the input data rate and resolution be decreased to make sure tracking is correct.

If your drive is configured to use one of the feedback types that follow, the controller divides the encoder data rate and resolution sent to the PSO by the value of the EmulatedQuadratureDivider Parameter.

Applicable Drive Types and Feedback Combinations (Primary Feedback)

  • Ndrive QL or QLe that uses Capacitance Sensor feedback
  • FireWire drive that uses -MXH feedback
  • Npaq that uses -MXR feedback
  • XC4e that uses an Encoder Multiplier with the -MX2 or -MX3 multiplier option
  • XR3 that uses an Encoder Multiplier with the -MX2 or -MX4 multiplier option

If your HyperWire drive is configured to use auxiliary feedback from an encoder multiplier with the -MX4 multiplier option, the controller divides the encoder data rate and resolution sent to the PSO by the value of the AuxiliaryEmulatedQuadratureDivider Parameter.

You can combine the operation of the AuxiliaryEmulatedQuadratureDivider or EmulatedQuadratureDivider parameter and the PSOTRACK SCALE command as necessary.

Maximum Quadrature Output Frequencies

On single-axis drives, the quadrature encoder signals can be echoed out of the drive and used as an input to the PSO hardware on another drive. This can be done for the signals and drives that follow:

  • Capacitance Sensor feedback on the Ndrive QL and QLe
  • Internal -MXH multiplier on the Ndrive HLe, HPe, and ML
  • XC4e that uses an Encoder Multiplier with primary feedback from the -MX2 or -MX3 multiplier option
  • Standard quadrature inputs on single-axis drives

If you are using an Npaq or an XR3 with one of the feedback options that follow, it is not necessary to externally correct the signals from the first axis to the next axis. But the output frequencies do have limitations. The applicable feedback options include:

  • Npaq with -MXR feedback
  • XR3 that uses an Encoder Multiplier with primary feedback from the -MX2 or -MX4 multiplier option
  • XR3 that uses an Encoder Multiplier with auxiliary feedback from the -MX4 multiplier option

The table that follows shows the output frequencies at which the controller can send quadrature encoder signals out of the drives. If the output frequency exceeds the maximum input tracking frequency of the PSO, use the EncoderDivider or the EmulatedQuadratureDivider parameter to decrease the frequency of the signals that the controller sends to the PSO without having an effect on overall system resolution.

If your HyperWire drive is configured to use auxiliary feedback from an encoder multiplier with the -MX4 multiplier option, use the EncoderDivider or the AuxiliaryEmulatedQuadratureDivider parameter.

PSO Latency Specifications

The PSO hardware requires a fixed quantity of time to respond to changes at the encoder inputs. As a result, a small latency occurs when you generate a fire event. These latencies vary with drive type and tracking mode.

When axis calibration is active for a given axis, additional logic is used in the PSO hardware to correct the incoming quadrature signals. In drives that support PSO calibration, the additional logic can add up to 40 nanoseconds of latency to the values given in the following table. PSO calibration is currently supported on all drives.

Jitter Specifications

A drive with -MXH or -MXR might generate emulated quadrature signals that are not exactly the same as the encoder position. When a drive with -MXH or -MXR samples a sine wave encoder and generates emulated quadrature signals, it can only give an integer number of counts per encoder sample, up to the maximum quadrature encoder output frequency. This can cause jitter between counts of generated emulated quadrature signals.

For example, if you are using an Ndrive HPe with -MXH and your encoder shows motion at 1.5 MHz, the drive will change between generating one count of emulated quadrature signals and then two counts of emulated quadrature signals per a 1 MHz encoder sample cycle.
