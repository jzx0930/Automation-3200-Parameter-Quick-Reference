Feedhold Overview

The FEEDHOLD command toggles the feedhold state of the controller. FEEDHOLD ON causes motion to immediately decelerate and FEEDHOLD OFF causes motion to accelerate back to speed. The ramp type is always linear for this acceleration and deceleration.

The FEEDHOLD command is logically ORed with the state of the hardware feedhold inputs (as defined by the FeedholdInput and FeedholdSetup parameters). The feedhold state will not release until all feedhold conditions are false.

The feedhold state stops execution of programs and immediate commands. A subsequent feedhold release resumes command execution.

The Feedhold Active bit of Task Status 0 will be on from the moment the motion begins to decelerate due to a feedhold until the moment the motion begins to accelerate back to speed due to a feedhold release. The Feedheld Axes Stopped bit of Task Status 1 will be on from the moment deceleration due to feedhold ends until the moment the motion begins to accelerate back to speed due to a feedhold release.

Asynchronous motions (MOVEABS, FREERUN, etc.) will not respond to feedhold requests. You can use OVERRIDEASYNC ON/OFF to have asynchronous motion respond to feedhold requests. Spindle motion only responds to feedhold requests if G101 is active. PVT and homing motions (refer to Homing Overview) never respond to feedhold requests. Slaved axes (refer to CAMSYNC command) do not respond to feedhold request directly but follow the motion of the master axis. If a RAPID command is executing when you issue a feedhold state, all axes will decelerate to a stop.

If a coordinated move command (LINEAR, CW, CCW, BEZIER) is executing when the feedhold state is issued, the axes decelerate to a stop while maintaining the programmed path.

The feedhold state will pause programmed motion on a task. By default, a task error occurs if you try to move axes that are feedheld by a task. Use the INTERRUPTMOTION ON/OFF Command instead of the FEEDHOLD command if you want to move axes after they have been feedheld by a task.
