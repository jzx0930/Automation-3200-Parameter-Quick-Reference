PSOTRACK INPUT Command

Configures the PSO distance tracking counters.

Syntax:

 
PSOTRACK <Axis> INPUT <Source1>
 
PSOTRACK <Axis> INPUT <Source1>, <Source2>
 
PSOTRACK <Axis> INPUT <Source1>, <Source2>, <Source3>

Example:

 
PSOTRACK X INPUT 2, 1, 0

Table:     PSOTRACK INPUT Command Arguments

Argument

Type

Description

<Axis>

Axis

The axis on which to execute the command.

<Source1>

32-bit signed integer

The encoder to use as the source for the first axis of tracking (use for single-axis).

<Source2>

32-bit signed integer

(Optional) The encoder to use as the source for the second axis of tracking (DO NOT use for single-axis).

<Source3>

32-bit signed integer

(Optional) The encoder to use as the source for the third axis of tracking (DO NOT use for single-axis).

Hardware: This command does not apply when used with the Nmark SSaM or the Nstep.

Hardware: The Ndrive CP, MP, and QL, the XC2, and the XC4, can do single-axis tracking only.

Hardware: The Ndrive HLe, HPe, and ML, the Npaq, Nservo, XC4e, and XR3 support multi-axis tracking only when you purchase the correct hardware option.

Use the PSOTRACK INPUT command to specify the feedback sources to be used for distance-based tracking. The number of <Source#> arguments that you specify for this command controls the number of axes that are tracked by the PSO hardware. You must specify the <Source1> argument. The <Source2> and <Source3> arguments are optional.

Because the PSODISTANCE command uses information that you specify in the PSOTRACK INPUT command, you must issue the PSOTRACK INPUT command before you issue the PSODISTANCE command.

The following sections show the possible values for the <Source#> argument for each drive type.

Warning: When you use this command to configure the functions of the Sync ports on the XC4 or XC4e or of the High-Speed Outputs on the XR3, the new functions of the outputs override all existing functions that you previously configured with other commands.

If the PositionFeedbackType parameter is set to Encoder Multiplier, the drive generates emulated quadrature signals only if the effective multiplication factor is greater than 4. The XR3 must use one of the multiplier options that follow:

  • For primary feedback, the XR3 uses an -MX2 or -MX4 multiplier option. The effective multiplication factor is equal to the value of the EncoderMultiplicationFactor parameter divided by the value of the EmulatedQuadratureDivider parameter.

The XR3 cannot use primary feedback from an -MX1 multiplier option to send emulated quadrature signals to the PSO.

PULSE command output (mode PULSEOUTPUTMODE_Null)

If you specify only a single <Source#> argument, the PSO hardware performs single axis distance tracking. In this mode, a PSO firing event will occur each time the distance moved by the source exceeds the specified firing distance.

If you specify two <Source#> arguments, the PSO hardware does two-axis distance tracking. In this mode, a PSO firing event will occur each time the sum of the squares of the two source counters exceeds the square of the firing distance. Another way of thinking about this is that a PSO firing event will occur any time the motion of the tracked axes intersect a circle with radius equal to the firing distance and whose center is at the location of the previous firing event.

If you specify three <Source#> arguments, the PSO hardware performs three-axis distance tracking. In this mode, a PSO firing event will occur each time the sum of the squares of the three source counters exceeds the square of the firing distance. Another way of thinking about this is that a PSO firing event will occur any time the motion of the tracked axes intersect a sphere with radius equal to the firing distance and whose center is at the location of the previous firing event.

You can see the PSO distance and window counters from the Drive Information Tab of the Status Utility. You can plot the PSO Counter 1, PSO Counter 2, PSO Counter 3, PSO Window 1, and PSO Window 2 items in Digital Scope.

Note: The PSO hardware operates on encoder counts and cannot do non-integer scaling. For true vector firing, all axes being tracked must have the same distance unit per encoder count.
