SYNCCALL Command

Executes a canned function at the same time as coordinated and RAPID motion.

Syntax:

 
SYNCCALL <Task> BLOCKING <ProgramName>, <Subroutine>
 
SYNCCALL <Task> NONBLOCKING <ProgramName>, <Subroutine>
 
SYNCCALL <Task> BLOCKING <ProgramName>, <Subroutine>, <ArgumentName> <ArgumentValue> [ <ArgumentName> <ArgumentValue> ... ]
 
SYNCCALL <Task> NONBLOCKING <ProgramName>, <Subroutine>, <ArgumentName> <ArgumentValue> [ <ArgumentName> <ArgumentValue> ... ]

Example:

 
SYNCCALL 2 BLOCKING "PROG2.PGM", "MYSUB", X3
SYNCCALL $taskindex NONBLOCKING "", "MYSUB"

Table:     SYNCCALL Command Arguments

Argument

Type

Description

<Task>

TaskID

The task on which to execute the subroutine. To execute the subroutine on the current task, use $taskindex as the value.

<ProgramName>

String

The name of the program to which the controller transfers program execution.

<Subroutine>

Label

The name of the subroutine to which the controller transfers program execution.

<ArgumentName>

Other

(Optional) The name of the argument that you want to pass to the subroutine.

<ArgumentValue>

64-bit IEEE floating point

(Optional) The value of the argument that you want to pass to the subroutine.

Use the SYNCCALL command to execute a canned function at the same time as coordinated (LINEAR, CW, CCW, BEZIER) and RAPID motion. After the coordinated motion command completes, the controller executes the canned function.

The <ProgramName> and <Subroutine> arguments can be Global String variables and define the program execution entry point. If you specify "" for the <ProgramName> argument, the controller uses the label in the calling program.

Note: Program names must be less than 187 characters long. Refer to Program Names for more information.

Note: Subroutine names must be less than 32 characters long. Refer to Subroutines Overview for more information.

Programs can call themselves or other programs. There is no limit to the nesting of calls, except for the limits from available memory and from the CallStackSize task parameter.

Before the controller executes a program called by the SYNCCALL command, you must download that program onto the controller. When the program is on the controller, it prevents an invalid file name or path error from occurring. Download the program at startup by using the Program Automation feature. Make sure that you set the Mode for the execution type to Download.

SYNCCALL Execution Modes

Use the BLOCKING or NONBLOCKING keyword to change the execution mode of the specified canned function. More information about the execution modes are in the sections that follow.

Blocking Mode

Use the BLOCKING keyword in the SYNCCALL command to execute the canned function as a blocking subroutine. When you do this, the controller does not execute the next motion command until the canned function completes. The controller will decrease the speed of motion to zero before it executes the canned function.

This mode is good for executing commands that could interfere with motion. The example that follows shows you how to use blocking mode.

Example

' Enable incremental programming mode.
INCREMENTAL
' Set the corner rounding axes to X and Y.
ROUNDING AXES X Y
' Set position tolerance of 0.5 units for corner rounding.
ROUNDING TOLERANCE 0.5
' Turn on rounding.
ROUNDING ON
' Move the X axis 11 units to the right.
' The next move makes a rounded corner.
LINEAR X11
' Execute a canned function to home the Z axis.
' The canned function executes after the previous LINEAR command.
SYNCCALL $taskindex BLOCKING "", "TriggerHome"
LINEAR Y11
' Turn off rounding.
ROUNDING OFF

END PROGRAM

' The canned function.
DFS TriggerHome
	HOME Z
ENDDFS

Non-Blocking Mode

Use the NONBLOCKING keyword in the SYNCCALL command to execute the canned function as a non-blocking subroutine. When you do this, the controller executes the canned function and then immediately continues to the next motion command without waiting for the canned function to complete.

This mode is good for executing commands during a VELOCITY ON sequence of moves without the controller decreasing motion to zero. The example that follows shows you how to use non-blocking mode.

Example

' Enable velocity blending mode.
VELOCITY ON
' Enable incremental programming mode.
INCREMENTAL
' Move the X axis 5.0 units.
LINEAR X5
' Move the X axis 10.0 units.
LINEAR X10
' Execute a canned function to freerun the Z axis.
' The canned function executes after the previous LINEAR command.
' The controller does not decelerate the X axis to zero velocity.
SYNCCALL $taskindex NONBLOCKING "", "TriggerFreerun"
' Move the X axis 5.0 units.
LINEAR X5
' Disable velocity blending mode.
VELOCITY OFF

END PROGRAM

' The canned function.
DFS TriggerFreerun
	FREERUN Z 100
ENDDFS
