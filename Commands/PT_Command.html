PT Command

Executes a time point move on the specified axes.

Syntax:

 
PT <Axis> <Position> [ <Axis> <Position> ... ] TIME <Time>

Example:

 
PT X 10 Y 15 TIME 1

Table:     PT Command Arguments

Argument

Type

Description

<Axis>

Axis

The axis on which to execute the command.

<Position>

64-bit IEEE floating point

If the task is in ABSOLUTE mode, this argument specifies the absolute position to move to, in user units. If the task is in INCREMENTAL mode, this argument specifies the distance to move, in user units.

<Time>

64-bit IEEE floating point

The total time of the move, in milliseconds.

Warning: To give more control to the programmer, this command removes some of the features that reduce jerk or unacceptable motion.

Note: To use the PT command, you must set the MotionInterpolationMode Parameter to 3-position / 1-velocity.

Use the PT command to manually enter profile points from a program to make arbitrary velocity or acceleration profiles for a single or multiple axes. This command bypasses all automatic acceleration, and acceleration limiting used in RAPID, LINEAR, CW, CCW, BEZIER, MOVEABS, MOVEINC, and FREERUN command moves.

The controller uses the same third order polynomial to determine the velocity command at each position point.

P is the position at the end of the time segment in user units. If the ABSOLUTE task mode is enabled, this value is entered as an absolute position. If the INCREMENTAL task mode is enabled, this value is entered as an incremental distance from the end of the last move.

T is the time interval in milliseconds. To specify a move duration of one millisecond or longer, enter any positive integer. To specify a move duration of less than a millisecond, enter 1/2, 1/4, 1/8, 1/16, 1/24, or 1/48. Do not enter any other fractional values for the time.

Note: If you specify a move duration of less than a millisecond for <Time>, all of the PT moves in that same sequence must use the same <Time>.

If you specify a move duration for <Time> that is greater than 1.0 divided by the position update rate of an axis, the controller interpolates motion to the position update rate. To interpolate motion, the controller uses the cubic-spline polynomial algorithm that you specified for the MotionInterpolationMode Parameter. For more information, refer to the table that follows.

PT commands do not behave like the other synchronous or asynchronous motion commands. After a PT command is processed, the program immediately executes the next line in the program, much like an asynchronous motion command (MOVEINC, MOVEABS, FREERUN). However, unlike an asynchronous motion command, the PT motion commands are buffered and executed one after another. If a stream of PT commands is executed sequentially, it is possible to fill this buffer. If the buffer is full, subsequent PT commands wait on the line until buffer space becomes available.

If you want your program to wait until PT motion is complete before it continues, you must issue the WAIT MOVEDONE Command or the WAIT INPOS Command. Make sure that you do not use the WAIT MODE MOVEDONE command or the WAIT MODE INPOS command because PT does not obey the setting of the mode commands. For more information, refer to Motion Done Overview.

PT does not obey Acceleration and Deceleration Ramping. When you specify positions in a sequence of PT commands, you must gradually change the axis positions from command to command to make smooth motion.

When the controller combines PT with LINEAR, CW, CCW, or BEZIER moves, the controller always decelerates to a stop on the LINEAR, CW, CCW, or BEZIER move that comes before the PT move. Similarly, a PT move that immediately comes before a LINEAR, CW, CCW, or BEZIER move must produce a zero velocity or the axis jerks.

Warning: To avoid velocity dropouts, enclose sequential PT commands in a CRITICAL START and CRITICAL END pair.

PT Limitations

  1. This command does not work with any transformation.
  2. The controller does not look ahead in the PT mode and does not slow down for corners.
  3. MFO has no effect on PT motion.
  4. MaxSpeedClamp parameter has no effect on PT motion.
  5. Feedhold does not decelerate in PT motion. It finishes the PT command and waits until the queue is flushed. Then motion abruptly stops.
