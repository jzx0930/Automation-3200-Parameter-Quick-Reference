Program Block Lookahead Overview

The controller uses lookahead to precalculate information for moves in your AeroBasic program. The table that follows shows the features that use lookahead and how the controller uses the precalculated information for each feature. The controller automatically activates lookahead when one or more of these features is on.

Table:     Features that Use Lookahead

Feature How the Controller Uses Lookahead

Corner Rounding

The controller uses the precalculated information to determine what corners must be rounded.

Cutter Radius Compensation

The controller uses precalculated moves to determine where to apply offsets for the cutter radius, and to prevent collisions on inside corners.

VELOCITY ON

The controller uses precalculated moves to determine the correct ending velocity for the current move. Then, the controller smoothly blends velocities between moves and safely decelerates when it is necessary.

To precalculate the necessary information, the controller executes program lines ahead of the move that is currently executing. The maximum number of program lines the controller can execute ahead is determined by the value you set for the MaxLookaheadMoves parameter.

Command Synchronization During Lookahead

When lookahead is active, some AeroBasic operations look like they occur out of order because the controller is executing program lines ahead of the currently executing motion. For example:

  • Because the controller assigns global variables and task variables during the pre-execution, the value of the variable is updated before the current move finishes executing. Thus, global variables and task variables can be used as move targets during a lookahead sequence.
  • I/O reads are not synchronized to motion during lookahead. An I/O read returns the value of the input during lookahead.

Some AeroBasic commands are always synchronized to motion and are synchronized when lookahead is active or not active. During the lookahead pre-execution, the AeroBasic program executes past the synchronized command, but does not execute the command until the motion before it is completed. For a list of commands that are always synchronized, refer to Motion-Synchronized Commands.

When lookahead is not active, all commands are synchronized to motion.

Restrictions to Commands the Controller Can Look Through

There are some commands that the controller cannot look through during lookahead. When the controller gets to one of these commands, lookahead temporarily stops and the controller does what follows.

  • During Corner Rounding, if the controller cannot look through a command and that command occurs between two moves that form a corner, the controller does not round the corner.
  • During Cutter Radius Compensation, if the controller cannot look through a command, it ends the move with cutter offsets that are perpendicular to the original move. On the next move, the controller tries to apply a correction arc to move the cutter to the correct position to start the next move. If the intersection of the two moves forms an outside corner, the correction works and the arc is applied. If the intersection of the two moves forms an inside corner, overcut occurs and the controller applies the correct offsets by blending them over the length of the second of the two moves. For this reason, Aerotech recommends that during a Cutter Radius Compensation, you do a lead-off move before you use any command in the list that follows.
  • During a VELOCITY ON sequence, if the controller cannot look through a command, it forces the motion to decelerate to zero velocity.

In the list that follows, the commands that apply only during a VELOCITY ON sequence are marked. All other commands in the list apply to Corner Rounding, Cutter Radius Compensation, and VELOCITY ON sequences.

Lookahead and VELOCITY ON Sequences

When you use lookahead during a VELOCITY ON sequence, some conditions cause the controller to force a deceleration either to a non-zero velocity or to zero velocity.

Non-zero Velocity

These conditions cause the controller to force a deceleration to a non-zero velocity.

  • A corner in the part geometry would cause an acceleration greater than the value of the CoordinatedAccelLimit or DependentCoordinatedAccelLimit parameters.
  • The acceleration around an arc would be greater than the value of the CoordinatedCircularAccelLimit parameter.
  • A move duration would be less than the value of the CoordinatedMoveDuration parameter.
  • Normalcy Concurrent Motion would cause the normalcy axis acceleration to be greater than the value of the DependentCoordinatedAccelLimit parameter.

Zero Velocity

These conditions cause the controller to force a deceleration to a zero velocity.

  • The controller must precalculate more moves than what is permitted by the value of MaxLookaheadMoves. This generates a task warning.
  • The controller must do Normalcy Alignment Moves during a VELOCITY ON sequence.
  • The CoordinatedAccelLimitSetup parameter is set to Decelerate to Zero Velocity, and the controller gets to a corner that would cause acceleration greater than the value of the CoordinatedAccelLimit parameter.
  • Before the controller executes Canned Cycles that are configured to run at the end of LINEAR moves.
  • For breakpoints and Step Mode.
  • The controller senses a change in dominance mode between two moves during a VELOCITY ON sequence. To change the dominance mode, use the G98 and G99 commands.

To Prevent Problems During Lookahead

Lookahead can cause unwanted slowdowns, overcuts, or performance problems if it is configured incorrectly, or if certain conditions occur in your AeroBasic program. To prevent these problems, obey these suggestions:

  • During a Cutter Radius Compensation sequence, do not use commands that force deceleration to zero velocity. To prevent overcut, do a lead-off move before you issue the command.
  • During a Corner Rounding sequence, do not use commands that force deceleration to zero velocity. This prevents incorrect rounding.
  • During a VELOCITY ON sequence, if you do not want deceleration to occur, do not use commands that force a deceleration to zero velocity.
  • Do not use feedrates that are higher than necessary. Use the F command to set your feedrate to a satisfactory speed. A feedrate that is too high causes the controller to precalculate more moves, which can cause performance degradation and consume extra computation time. It can also cause the controller to be limited by the MaxLookaheadMoves parameter and cause one or more unwanted decelerations to zero velocity.
  • Do not use ramp rates that are lower than necessary. A ramp rate that is too low causes the controller to precalculate more moves, which can cause performance degradation and consume extra computation time. It can also cause the controller to be limited by the MaxLookaheadMoves parameter and cause one or more unwanted decelerations to zero velocity.
  • Avoid putting many non-motion program lines in between moves in your AeroBasic program if these moves are in a Cutter Radius Compensation, Corner Rounding, or VELOCITY ON sequence. Instead, do large sections of non-motion actions outside of lookahead sequences by issuing the G40, ROUNDING OFF, or VELOCITY OFF commands prior to the section of non-motion actions. If terminating lookahead is not possible, move the non-motion section into a subroutine and call the subroutine using the SYNCCALL command. Putting many non-motion lines in between moves in a lookahead sequence can lead to performance degradation and consume extra computation time. Doing so can also cause the controller to be limited by the MaxLookaheadMoves parameter, resulting in unwanted slowdowns.
  • When you are running a program in queue mode, make sure that there are enough downloaded program lines in the queue for lookahead to precalculate enough moves. If there are not enough moves in the queue, the controller forces a deceleration to zero velocity. Refer to Queue Mode Overview for more information.
  • Follow the steps in the Optimizing Performance in VELOCITY ON Mode topic.
